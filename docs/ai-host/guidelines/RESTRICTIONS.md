# 🚫 開発および行動に関する制約事項 (Restrictions)

AIアシスタントは、以下の制約事項を遵守して開発・提案を行う必要があります。

### 1. 破壊的変更の禁止
*   **データの消失**: 既存のデータベース(`dev.db`など)を削除・初期化するコマンドは、ユーザーの明示的な許可がない限り実行禁止。
*   **ファイルの削除**: ソースコードやドキュメントを削除する際は、必ずバックアップやGitの状況を確認すること。

### 2. デザイン・UI/UXの制約
*   **変更の禁止**: 既存のデザインシステム（Tailwindの設定、色の組み合わせなど）を独断で変更しないこと。
*   **一貫性**: 新しいUIコンポーネントを追加する際は、既存のコンポーネントのデザインパターンを継承すること。

### 3. コミット・Git操作の制約
*   **自動コミットの禁止**: ファイルを変更した後、ユーザーの確認なく `git commit` を行わないこと。必ず変更内容を提示し、許可を得てからコミットすること。
*   **コミットメッセージ**: 日本語ではなく、英語のConventional Commits形式（`feat:`, `fix:`, `docs:` 等）を使用すること。

### 4. ドキュメントの言語
*   **基本言語**: ドキュメント（Markdownファイル）は原則として **日本語** で記述すること。
    *   例外: `README.md` の一部や、あえて英語で記述することが規定されているファイル。
    *   例外: コード内のコメントや識別子、コミットメッセージは英語とする。

### 5. パッケージ・ライブラリの追加
*   **承認必須**: 新しいnpmパッケージを追加する際は、その必要性と代替案を提示し、ユーザーの承認を得ること。

### 6. 機能拡張の禁止 (Scope Creep Prevention)
*   **合意形成の必須化**: Phase 8 以降の機能追加、およびロードマップにない新機能の実装は、**事前にユーザーとの合意形成（Consensus）が完了するまで着手してはならない**。
*   **Generic Base Kitの維持**: 特定の個人や業種に特化しすぎる機能（ハードコードされた固有名詞など）の実装を避け、常に「配布可能な汎用版」としての品質を維持すること。

### 7. プロセス制御の指針 (Process Control)
*   **セッション終了の待機**: セッションハンドオーバー（終了処理）は、ユーザーから明確な指示（`/session_handover` や「終了」の合図）があるまで自動的に開始せず、次の指示を待つこと。**「すべてのタスク完了」は終了のトリガーではない。**
*   **フェーズ移行時の確認**: 新しいフェーズや、タスクの大きな区切り（新しいドキュメントの作成を含む）に取り掛かる際は、事前に「次はこのタスクを行いますがよろしいですか？」と確認し、ワンクッション置くこと。（※通常のコーディング中の細かな修正はこの限りではない）
