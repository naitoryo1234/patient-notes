# 設計原則 (Design Principles)

このドキュメントは、本プロジェクトにおける設計判断の基本方針を定めます。
新機能の設計や既存機能の改修時に参照してください。

---

## 1. 記録の透明性（Audit Trail）

**原則**: すべての記録には「誰が」「いつ」作成・更新したかを明示する

### 理由
- 業種を問わず、記録システムの基本仕様である
- **責任の明確化**: 誰がこの情報を残したかが分かる
- **情報の鮮度**: いつ時点の情報かが分かる
- **チーム協働**: 複数人で同じ案件を扱う際の基盤となる
- **法的・コンプライアンス対応**: 監査や紛争時の証拠能力

### 適用例
| 機能 | 適用 |
|:---|:---|
| カルテ記録 | 記録者（createdBy）と記録日時（createdAt）を常に表示 |
| 申し送り解決 | 解決者と解決日時を表示 |
| 予約作成/更新 | createdBy/updatedByをDBに記録 |

### 注意点
- 「担当者と記録者が同じ場合は省略」のような最適化は避ける
- 冗長に見えても、一貫して表示することで信頼性が担保される

---

## 2. 汎用性の考慮

**原則**: 特定業種（鍼灸院など）に最適化しすぎず、一般的なビジネス記録システムとして成立する設計を心がける

### 理由
- 将来的な用途拡張の可能性
- 設計判断の普遍性（業種依存の判断は脆い）

### 適用例
- 「医療記録だから必要」ではなく「記録システムとして当然」という視点で判断
- 専門用語よりも汎用的な表現を優先（設定で切替可能にする）

---

## 3. 明示性 over 推測

**原則**: ユーザーが推測しなくても分かるUIを目指す

### 理由
- 「表示がない = デフォルト値」という暗黙ルールは混乱を招く
- 情報が多少冗長でも、明示されている方が信頼できる

### 適用例
- 担当者と記録者が同じでも両方表示
- 未設定の項目は「未設定」と明示（空欄にしない）
