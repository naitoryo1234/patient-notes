# システム構造の再整理と汎用化の方針

## 1. 予約管理台帳 (`/appointments`) の左カラムについて
**決定: 「本日の予定」パネルは維持する**

### 理由
*   予約管理（電話対応や日程調整）を行いつつ、当日の混雑状況やスタッフの動きを「あえて画面遷移せずに」確認したいニーズが高いため。
*   ダッシュボード（現場対応）と台帳（計画管理）で機能は重複するが、**「コンテキストスイッチ（画面を行き来するコスト）を下げる」** というUX上のメリットを優先する。

### 今後の整合性担保
*   UIコンポーネントの共通化を徹底し、ロジック（アラート条件など）が分散しないように注意する（※直近の実装で `unresolvedMemos` のフェッチロジックなどは統一済み）。

## 2. 運用シナリオの汎用化 (Generalization)

`docs/specs/OPERATIONAL_SCENARIOS.md` を「鍼灸・整体」特化から、汎用的な「予約・顧客管理ベースキット」として書き換えます。

### 変更の方向性
*   **主体**: 「患者」→「顧客 (Client/Customer)」、「施術者」→「スタッフ (Staff)」
*   **行為**: 「診療/施術」→「サービス/対応 (Service)」
*   **具体例の抽象化**:
    *   「リハビリで再来院」→「同日内の複数回利用（コース連続、家族利用など）」
    *   「急患」→「緊急対応 / イレギュラー」

この方針に基づき、ドキュメントを更新します。
